# PRD Generator

Produce a product requirements document from a feature request or problem
statement. No technical details in the output — the goal is a document that a
product owner can review and approve without engineering knowledge. Claude uses
full codebase and architecture awareness behind the scenes to split stories
vertically and ground requirements in what's realistically achievable.

**Prompt style:** follow `${CLAUDE_PLUGIN_ROOT}/references/multi-choice.md` for
any question whose answer space is enumerable — work type (bug / feature /
technical / epic), persona, success-metric shape, scope cut-lines. Always show
a recommended choice; accept multi-select when the question is additive
("which personas: 1, 3"). Free text only for names and genuinely open
brainstorming.

**Plain language & trust:** apply
`${CLAUDE_PLUGIN_ROOT}/references/plain-language.md` to every question,
recommendation, and summary you show the user — plain wording, a stated reason
behind each recommendation, and technical detail only when the user asks. (The
PRD output stays non-technical regardless.)

## Step 0: Surface prior learnings

Before asking the user anything, read `docs/learnings/INDEX.md` in the target
repo. If the file does not exist, skip silently.

Filter entries relevant to the user's request:

- `description` or `name` contains keywords from the feature name / problem
  statement
- Entry type ∈ {`convention`, `pattern`, `win`} — these shape approach
- Entry type = `blocker` — warns off dead ends

If ≥1 relevant entry is found, surface to the user before the first question:

```
I found relevant prior learnings in this repo:
  - [title] ([type]) — [one-line description]
  - [title] ([type]) — [one-line description]

These will inform my suggestions. Anything to correct before we start?
```

Accept a bare "looks good" / "continue" as confirmation. If the user flags
something, read the full learning file(s) before moving on.

If no entries match or the index is missing, continue silently.

## Step 1: Extract Product Details

Scan the conversation for:

- **Feature name** — a short descriptive name (kebab-case for file naming)
- **Ticket/issue number** — ask if not mentioned; accept `TBD` if unknown
- **Scope** — bug / small enhancement (< 1 day) vs. single feature vs. multi-story epic
- **Nature** — bug / user-facing (has an end user) / technical (refactor, infra, dependency upgrade, performance, security, migration, tech debt — no end-user story). Infer this from the description — do not ask the user to classify it. If someone describes upgrading a library, that's technical. If they describe a new user-facing capability, that's user-facing.
- **Target user** (for user-facing) or **affected system** (for technical)
- **Business problem** — what is wrong, missing, or needed

## Step 2: Prerequisites Gate

Before proceeding, confirm you have enough context:

- What the feature/fix does and the problem it solves
- Who it's for and why they need it (or what system and why, for technical tasks)
- How success will be measured
- What is explicitly in and out of scope

If any of these is unclear, ask the user before loading context or writing anything.

## Step 3: Context Loading

Load full codebase context to produce better-structured PRDs:

1. Read this repo's `CLAUDE.md` to understand the project domain, architecture,
   user personas, and conventions. If it has a `## Learnings` section, those rules may shape the feature's acceptance criteria and scope boundaries — scan for `convention-` and `pattern-` entries relevant to the feature area.
2. Read `docs/architecture.md` if it exists (generated by `/doc-architect`).
3. Check `docs/discovery/` for existing discovery briefs related to the feature
   area. If a brief exists with `status: complete`:
   - Pre-load the desired outcome, selected opportunity, and solution candidates
     as context for this PRD.
   - Use the brief's solution candidates as starting points for story
     identification (for epics). The PRD owns the final story split based on
     codebase understanding, but discovery provides strategic guidance.
   - Note the brief path — you will reference it in the PRD output.
4. Grep for existing implementations related to the feature area — understand
   system boundaries to inform vertical story slicing.
5. Check existing specs in `docs/specs/` for overlap or dependencies.

**Purpose**: Use architecture and code understanding to split stories along real
system boundaries, determine the right number of stories, and ground the PRD in
what's realistically achievable. The PRD _output_ remains non-technical, but the
_process_ is fully informed by the codebase. When a discovery brief exists, it
provides the strategic rationale (outcome, opportunity, solution direction) that
grounds the PRD in validated product thinking.

## Step 4: Product Interrogation Gate

Now that the codebase and architecture have been loaded, **compose a draft of the
requirements** from the conversation and loaded context (the user stories,
business rules, scope, and success criteria you intend to write), then
**self-grill that draft** following
`${CLAUDE_PLUGIN_ROOT}/references/self-grill.md`: critique it for gaps,
contradictions, and unstated assumptions; resolve the minor ones into an
assumptions summary you'll fold into the PRD; and surface only the genuine
product decisions, one at a time. Codebase awareness allows sharper, more
grounded questions (e.g. "this feature touches both registration and
notifications — should we split those into separate stories?"). This replaces the
old manual `grill-me` pass — don't invoke the standalone `/grill-me`.

**Restrict all questions to product and business concerns** — the audience is a
product owner, not an engineer.

Ask about:

- User needs, personas, and workflows
- Business rules and edge cases
- Success criteria and metrics
- Scope boundaries and trade-offs (informed by system boundaries)
- Story decomposition and vertical slicing (informed by architecture)
- Rollout risks and dependencies
- Impact and priority

Do NOT ask about:

- Architecture, APIs, database design, or code structure
- Technical implementation choices
- File paths, endpoints, or data models
- Performance benchmarks or infrastructure topology

## Step 5: Choose Template

| Scope / Nature               | Template(s) to read                                                                                                |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Bug report                   | `${CLAUDE_SKILL_DIR}/references/template-bug.md`                                                                   |
| Small enhancement (< 1 day)  | `${CLAUDE_SKILL_DIR}/references/template-simple.md`                                                                |
| Single feature (user-facing) | `${CLAUDE_SKILL_DIR}/references/template-feature.md`                                                               |
| Single task (technical)      | `${CLAUDE_SKILL_DIR}/references/template-technical.md`                                                             |
| Multi-story epic             | `${CLAUDE_SKILL_DIR}/references/template-epic.md` + one `template-feature.md` or `template-technical.md` per story |

Read the appropriate template file(s) before generating the PRD. The template
defines the output format — fill in its sections with content specific to this
feature.

Also read `${CLAUDE_SKILL_DIR}/references/bdd-format.md` for Gherkin
acceptance criteria format rules.

**For multi-story epics**, identify each distinct story first:

- Each story must deliver independent user value and map to exactly one ticket
- Split stories vertically based on codebase understanding of system boundaries
- User-facing stories use "As a [user], I want to [action] so that [benefit]"
- Technical stories use the technical template (no forced user story)
- Generate one epic overview spec + one story spec per story

## Step 6: Fill the Template

Write spec file(s) to `docs/specs/{ticket}-{name}/` at the repo root, creating
the `docs/specs/` directory if it doesn't exist. Use just `{name}/` if there's
no ticket number.

File naming:

- Single story/bug/enhancement: `spec.md`
- Multi-story epic: `spec.md` (overview) + `spec-{story-slug}.md` per story

### Multi-story epics: agent-based story writing

For epics with multiple stories, write the PRDs in two phases:

**Phase A — Write the epic overview yourself.** Fill in `spec.md` using
`template-epic.md`. This establishes: story boundaries, shared business rules,
the user journey map, and the Story Index with file paths for each story spec.
The overview must be complete before any story specs are written.

**Phase B — Spawn `prd-story-writer` agents in parallel**, one per story.
Each agent receives:

- The epic overview spec path (so it can read shared context and story boundaries)
- The assigned story name, ticket, and type (user-facing or technical)
- The output file path (`docs/specs/{ticket}-{name}/spec-{story-slug}.md`)
- The template to use (`template-feature.md` for user-facing, `template-technical.md` for technical)
- The path to `bdd-format.md`

Launch all story agents in parallel. Wait for all to complete, then:

1. **Read each story spec** — verify consistency with the epic overview. Fix any
   gaps, overlaps, or contradictions.
2. **Update the epic overview** — ensure the Story Index in `spec.md` has correct
   links to every written story spec file. Update ticket numbers, story names, and
   dependency relationships if any were resolved during writing.

### Quality rules

Apply these rules while filling in templates (and verify agent output against them):

- **No placeholders** — never leave `[TBD]`, `[placeholder]`, `TODO`, or
  `[Insert X]` markers (ticket IDs may be `TBD`)
- **No technical content in output** — no file paths, API endpoints, database
  tables, error codes, HTTP status codes, or technical jargon anywhere in the
  PRD. The output must be readable by a non-technical product owner.
- **User stories where required** — user-facing features and enhancements MUST
  include "As a [user], I want to [action] so that [benefit]". Bugs and
  technical tasks do NOT include a user story.
- **Concrete examples** — use realistic business scenarios with real-world
  values like names, dates, and quantities. At least 3 concrete examples
  across the PRD. For bug reports, reproduction steps with specific values
  count as examples — no need to add separate ones.
- **Thorough acceptance criteria** — Gherkin Given/When/Then from the user's
  perspective. Cover ALL happy paths, error cases, edge cases, and business
  rules. Do not artificially cap the number of scenarios.
  Read `bdd-format.md` for format rules.
- **Vertical story slicing** — for epics, split stories along system boundaries
  informed by codebase understanding. Each story must deliver independent value.
- **Delete unused sections** — if a section doesn't apply, delete it entirely.
  No empty stubs.
- **Discovery brief backlink** — if a discovery brief was loaded in Step 3, add
  a `**Discovery Brief:** docs/discovery/{name}/brief.md` reference near the top
  of the PRD (after the title). For epics, also update the discovery brief's
  Solution Candidates table with the PRD spec path.

## Step 7: Validate

Before presenting the PRD, verify every item below. Fix any failures.

- [ ] No `[TBD]`, `[placeholder]`, `TODO`, or `[Insert X]` markers (ticket IDs may be TBD)
- [ ] No file paths, API endpoints, database tables, error codes, or technical jargon anywhere
- [ ] At least 3 concrete business examples with realistic values
- [ ] User-facing features and enhancements have an "As a..." user story
- [ ] Bugs do NOT have a user story
- [ ] Technical tasks do NOT have a user story and have a `**Type:** Technical — [subtype]` label
- [ ] All acceptance criteria use Gherkin Given/When/Then from the user's perspective
- [ ] Acceptance criteria are thorough — all happy paths, errors, edge cases, and business rules covered
- [ ] Scope section has both "In scope" and "Out of scope"
- [ ] Success metrics are measurable from a business/product standpoint
- [ ] Optional sections deleted if not applicable — no empty stubs
- [ ] For epics: each story delivers independent user value
- [ ] For epics: stories are split vertically based on system boundaries
- [ ] Open Questions resolved or explicitly deferred with rationale

## Step 8: Jira Integration (Optional)

Ask the user: **"Would you like to create Jira issues for this PRD?"**

If yes:

1. **Identify the project key** — check `CLAUDE.md` for a configured Jira project key (e.g. a line like `Jira project: PROJ`). If not found, ask the user: "Which Jira project key should I use? (e.g. `PROJ`, `ENG`)"

2. **Create Jira issue(s)** using the `mcp__plugin_atlassian_atlassian__createJiraIssue` tool, based on PRD type:

   | PRD Type           | Jira issue type  | What to create                                       |
   | ------------------ | ---------------- | ---------------------------------------------------- |
   | Bug report         | `Bug`            | One Bug ticket                                       |
   | Simple enhancement | `Story`          | One Story                                            |
   | Single feature     | `Story`          | One Story                                            |
   | Technical task     | `Story`          | One Story                                            |
   | Multi-story epic   | `Epic` + `Story` | Create the Epic first, then one Story per story spec |

   For each issue:
   - **Summary**: Use the PRD title (strip markdown formatting)
   - **Description**: Use the PRD summary paragraph (2-3 sentences, plain text)
   - For multi-story epics: create the Epic first, then create each Story with the Epic's issue key as the parent/epic link

3. **Update spec files** — replace placeholder ticket values with the real Jira issue key returned from each `createJiraIssue` call:
   - Single spec: update `**Ticket:** TBD` → `**Ticket:** PROJECT-123`
   - Epic `spec.md`: update the Story Index table with the real Epic key and each Story key
   - Each story spec: update `**Ticket:** TBD` → the real story ticket key
   - Rename the `docs/specs/{ticket}-{name}/` directory if it was created with `TBD` as the ticket prefix

4. Report the created Jira issue keys and URLs to the user.

If the user declines Jira integration, skip this step entirely.

## Step 9: Handoff

Then STOP. Tell the user:

1. The path(s) of the spec file(s) written
2. Any Jira issues created (keys and URLs) — skip if Step 8 was skipped
3. Ask for approval before handing off to engineering
